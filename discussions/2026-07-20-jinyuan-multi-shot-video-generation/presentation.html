Literature report · 2026.07

这份报告讨论 multi-shot video generation。重点不是逐篇罗列论文，而是回答一个共同问题：多个镜头在生成时怎样共享信息。我们会沿四条路线展开，从一次联合去噪、逐镜头生成、因果流式生成，一直到模型外部的规划与反馈系统。

贯穿全场的判断是：multi-shot 不是把几段短视频拼在一起，而是要在 hard cut 制造变化的同时，控制人物、道具、环境和叙事状态中哪些保持、哪些改变。

≠ Long video

long video 强调持续时间；multi-shot 强调镜头边界与镜头间依赖。

≠ Multi-scene

scene 是叙事时空单位，一个 scene 可以包含多个 shots。

≠ Story video

本报告限定为显式建模 shot boundary 与 shot-level dependency 的视频方法。

核心难点：consistency
不是所有镜头都相似，而是“该保持的保持、该变化的变化”。
Boundary: model chunk 只有与语义镜头边界对应时，才算 shot

先给出任务定义：输入可以是完整故事，也可以是全局设定加逐镜头描述；输出是一段由明确 cut 分隔的视频。镜头之间允许改变视角、构图、动作甚至时空，但共同表达连贯叙事。

它不等于单纯的长视频，也不等于多场景。最重要的难点是 consistency：hard cut 本来就要求变化，所以指标不能只奖励相似，而要判断人物身份、道具状态、环境、风格和因果关系中哪些应当延续。

Taxonomy follows inference interface, not module names

这页同时承担 taxonomy 和目录。Holistic Joint 在一次调用里让所有镜头双向交互；Shot-Level AR 多次调用模型，每次产出完整镜头；Causal Streaming 进一步把生成粒度降到 frame 或 chunk；System 路线则在模型外部规划、检查与纠错。

分类依据是主要推理接口，不是论文中出现的模块名称。接下来依次进入四个 Part，最后用三页综合比较 trade-off、benchmark 现状和开放讨论。

Holistic Joint 的共同特征，是在推理前就确定镜头数量、边界或时长，并准备逐镜头 caption；这些 caption 可以由 LLM 从故事自动解析，但在调用视频模型时已经存在。所有视觉 token 在同一次去噪中双向交互，所以没有逐轮 rollout 的误差累积。

真正困难的是同一个上下文里有多条 caption：怎样让第 i 个镜头主要执行第 i 条描述，同时保留跨镜头视觉交流。这就是接下来三页的 caption-shot alignment。

第一类方法直接设计 attention mask。Mask²DiT 让第 i 个 shot 只看到第 i 条 caption，避免文本串到错误镜头；与此同时，视觉 token 在不同 shots 之间仍然互相可见，用来保持人物与风格一致。

CineTrans 只在部分层切断跨 shot 视觉注意力，让这些层专门学习 hard cut，其余层保留全局交流。ShotDirector 的 Global–Local mask 则让人物和故事条件全局可见，而摄影与相机描述只作用于所属镜头。

第二类方法不直接切断 attention，而是修改位置编码。问题来自每个镜头的局部时间都可能从零开始：模型既要知道 shot 内的连续时间，也要知道 shot 的顺序和对应 caption。

第三类是 transition token。ShotAdapter 为 K 个镜头插入 K−1 个共享 token 副本，给模型一个显式的“这里应该切镜”的控制手柄。但要注意分工：duration 和 local mask 决定切点在哪里，transition token 强化切镜这个操作本身。

论文消融显示，即使去掉 transition token，只保留 local mask，模型仍能生成多镜头视频。所以它不是 multi-shot 能力的唯一来源。它的价值是把 cut 变成可学习、可操控的信号，但还没有细分 shot-reverse-shot 等复杂剪辑类型。

shot 内保持 dense attention；跨 shot 只用代表 / summary tokens 双向通信，降低全序列平方开销。

Window Cross-Attention 继续解决 per-shot prompt alignment，但不是相对前三页最主要的新贡献。

Source: HoloCine paper, Figure 2

HoloCine 的位置不是提出第四种 caption alignment 原语，而是把一次联合生成扩展到更长时间。论文训练数据覆盖约 5、15 和 60 秒，最长训练样本最多约 13 个镜头。这里应该理解为训练覆盖范围，不是推理硬上限。

关键机制是 sparse inter-shot self-attention：镜头内部仍密集建模，跨镜头只通过代表或 summary token 双向沟通，从而避免全序列 dense attention 的平方成本。代价是仍需完整脚本，而且细粒度远程状态可能在摘要通信中丢失。

Shot-Level AR 把完整镜头作为自回归单位。每轮先从已经完成的历史中选择或压缩信息，再用一次完整 diffusion sampling 生成当前镜头，最后把结果写回 memory。

它和 holistic 的区别是过去一旦生成就固定，未来内容无法反向修正；和 causal streaming 的区别是镜头内部仍然可以使用双向注意力，通常要等整次采样完成才产出镜头。演进重点不是简单堆更多历史帧，而是怎样组织、检索、更新和验证 memory。

OneStory 的清晰贡献是把 multi-shot 改写为 next-shot generation，同时允许当前 caption 从全部历史中选择证据。Frame Selection 用 caption 查询历史 latent frames，避免机械地只看最近镜头；Adaptive Conditioner 再按重要性压缩选中上下文。

压缩后的 memory tokens 与当前噪声 token 一起进入生成器，所以历史范围可以是全局的，但条件预算仍然紧凑。它支持逐轮追加 shot caption，不过不负责自动规划故事，而且任何历史错误都会成为后续检索的错误证据。

FilmWeaver 的主要洞见是把两种时间尺度拆开。Shot Cache 从全部旧镜头中按当前 prompt 检索关键帧，承担人物与背景的长程召回；Temporal Cache 保存当前镜头的近期帧，承担动作连续和镜头延展。

这两类 cache 作为 in-context 条件进入同一 diffusion model，并覆盖文本到视频、新镜头、镜头延展等模式。它说明跨镜头 identity 与镜头内 motion 不应该用同一种记忆解决。但“任意长度”只代表接口可扩展，不能等同于无限 rollout 不漂移。

Causal Streaming 的严格定义包含三点：镜头内部继续按 frame 或 latent chunk 因果 rollout；历史以有界 KV、latent state 或参考帧复用；新 prompt 可以在生成过程中到达，系统无需重算已经输出的内容。

因此，仅仅多次调用模型、每次完成一个镜头，不属于这一路线。它的优势是低首帧延迟、可任意追加和运行时交互，代价是过去不能被未来反向修正，误差会沿 causal state 持续传播。

System is orthogonal to the renderer architecture

System 路线不直接规定视频模型内部怎样生成，而是在模型之外决定怎样拆分故事、按什么顺序生成、保存哪些参考，以及什么时候重试或修改后续输入。它可以调用 joint、shot-AR 或 causal renderer。

这里不按 agent 数量或模块名称分阶段，而是看实际生成结果是否会影响后续决策。由此得到三个清晰阶段：预先规划与单向执行；生成过程中的候选选择和外部状态管理；以及视觉反馈真正改变镜头顺序、提示词、memory 或重试策略的闭环。

GroundShot 的关键不是多加几个 agent，而是改变镜头生成顺序。它不机械地按故事顺序执行，而是先找到最适合作为实体参考的镜头，生成并验证；只有参考可靠才写入 memory，再服务其他镜头。

没有单一 SOTA：任务接口决定合理架构。

Choose architecture after fixing interaction and editing requirements

统一比较后可以看到，没有单一路线在所有维度胜出。Holistic 最擅长固定脚本下的全局双向协调，但首帧延迟高、局部修改代价大。Shot-AR 能追加并重用完整双向生成器，但会受到 memory drift。Causal Streaming 延迟最低，却承担持续误差累积。

Agent system 可以重排、检查和重做，但成本与 verifier 偏差不可忽视。因此讨论 SOTA 之前，必须先固定用户是否要求在线追加、是否需要局部编辑、允许多少重试和外部 API。
